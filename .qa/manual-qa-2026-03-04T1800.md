## Bugs Found

### BUG-1: Slug conflict on import leaks raw DB error (TC-17) — FIXED
**Input**: Import entity with same slug as existing entity but different UUID
**Was**: Raw DB error: `UNIQUE constraint failed: entities.slug`
**Fix**: Catch `sqlx::Error::Database` with slug constraint message, wrap as `FilamentError::Validation`
**After**: `error: Validation: import conflict: slug 't63pw55j' is already used by a different entity` (exit 4)
**Regression test**: `import_slug_conflict_returns_validation_error`

### BUG-2: FK violation on import leaks raw DB error (TC-20) — FIXED
**Input**: Import relation referencing non-existent source/target entities
**Was**: Raw DB error: `FOREIGN KEY constraint failed`
**Fix**: Catch `sqlx::Error::Database` with FK constraint message, wrap as `FilamentError::Validation` with entity IDs
**After**: `error: Validation: import error: relation '...' references non-existent entity (source: ..., target: ...)` (exit 4)
**Regression test**: `import_relation_with_missing_entity_returns_validation_error`

## Observations (not bugs)

1. **Escalation display can't be tested via CLI alone** (TC-23, TC-24 = N/A): Escalations come from (a) unread messages TO "user" from agents, and (b) agent_runs with status "blocked"/"needs_input". Both require daemon mode or direct DB manipulation. The escalation *command* works correctly; we just can't generate test data through normal CLI commands.

2. **Context --json returns formatted strings, not structured data** (TC-27): The JSON output is `["[module] auth-mod: Authentication module", ...]` — an array of display strings. For programmatic agent use, structured objects with separate fields would be more useful. Low priority enhancement.

3. **Import counts are technically "upserted" not "imported"** (TC-07-09): Re-importing entities shows `entities: 4` (because ON CONFLICT DO UPDATE always counts), but `relations: 0, messages: 0, events: 0` (INSERT OR IGNORE skips duplicates). Behavior is correct but the count semantics differ between entity types.
