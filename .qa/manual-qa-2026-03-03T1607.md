## Findings

### Finding 1: Inconsistent `exclusive` default (Low severity)
- **CLI** `reserve` defaults `--exclusive` to `false` (shared mode)
- **Daemon handler** defaults `exclusive` to `true` via `unwrap_or(true)`
- Agents using CLI get non-exclusive; agents using daemon socket get exclusive
- **Recommendation**: Align defaults — either both `true` or both `false`

### Finding 2: PID file brief race on daemon startup (Cosmetic)
- TC-03 ran immediately after `serve` and found no PID file yet (socket existed)
- TC-03b with 1s delay found the PID file
- Root cause: background re-exec, child hasn't finished init
- Not a practical issue — double-start check uses `is_process_alive()`

### Finding 3: CLI does not route through daemon (By design)
- When daemon is running, CLI refuses direct SQLite access with clear error
- CLI-to-daemon routing is a future enhancement (Phase 4+)
- Error message is good: "Stop the daemon with `filament stop` or use a daemon-aware client"

### Finding 4: `task assign` requires existing agent entity (By design)
- `filament task assign --to agent-a deploy-staging` → "Entity not found: agent-a"
- Must first create: `filament add --type agent agent-a --summary "..."`
- Consistent with graph model where all nodes must exist before creating edges

## Bugs Found

| # | Severity | Description | Status |
|---|----------|-------------|--------|
| 1 | Low | `exclusive` default mismatch between CLI (false) and daemon (true) | **Fixed** — daemon now defaults to `false` to match CLI |

No high-severity bugs found. All error paths produce clear messages with hints and correct exit codes.
